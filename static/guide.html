Architecture

Forge MES is organized around a simple event-driven core:

Operator actions and APIs -> MES database -> immutable event log -> SHA-256 hashing -> blockchain anchor service -> verification API -> operator and agent workflows

The backend is FastAPI with SQLModel entities, a service layer for business rules, WebSocket updates for live audit and equipment state, and MCP-compatible agent endpoints. The frontend is a lightweight operator workspace focused on batch execution, traceability, and verification.

The key architectural boundary is that the blockchain layer does not own MES state. The database remains the source of truth, while the blockchain anchor layer independently proves whether critical records have changed.

Core Components

Batch management: create, start, execute, complete, and inspect batches.

Recipe management: versioned recipes with approval and signatures.

Electronic batch record: append-only events with timestamps, users, comments, and signatures.

Material genealogy: lot relationships and batch linkage.

Equipment and OEE: runtime, downtime, throughput, and quality tracking.

Agent workflow: next-best actions derived from live MES context.

Blockchain Component

The blockchain layer is a verification layer, not the main storage system. Forge stores the actual MES records in the database and anchors only a SHA-256 hash of the approved recipe version or completed batch record.

What gets anchored: approved recipe versions and completed electronic batch records.

What gets stored on the blockchain side: entity type, entity id, hash value, tx id, timestamp, and lightweight metadata.

Where Forge stores the blockchain evidence: the MES keeps anchor evidence in its blockchain anchor table, alongside the batch and recipe tables, so verification can compare the current database record against the original anchored hash.

Why this is immutable: the tx id and stored hash become the frozen reference point. If the underlying record changes, the recalculated hash diverges from the stored anchored hash and Forge reports the mismatch.

Why this matters: if someone modifies the record after anchoring, the new hash no longer matches the stored anchor, so tampering becomes detectable.

Extraction path: operators can inspect anchors through the verification API or export an eBR PDF that includes the tx id, stored hash, and verification result.

ALCOA++ Coverage

Attributable: user identity and electronic signature are attached to critical actions.

Legible: every record is readable in the UI, API, and PDF export.

Contemporaneous: events are timestamped at creation and appended immediately to the eBR.

Original: event records are append-only, and completed records are hash-anchored.

Accurate: verification recomputes the record hash to expose tampering.

Complete: recipe, batch, genealogy, telemetry, and signatures are linked in one flow.

Consistent: the service layer applies the same transitions and audit rules across APIs and UI.

Enduring: the eBR persists in the database and can be exported to PDF.

Available: REST, WebSocket, MCP, UI, and PDF export all expose the same evidence chain.

How to Verify

1. Complete a batch.

2. Click Verify Anchor in the operator workspace.

3. Forge recalculates the current batch hash.

4. If the recalculated hash equals the stored anchored hash, the record verifies.

5. If they differ, Forge reports an integrity mismatch.

How to Extract Blockchain Evidence

Use the batch PDF export to download the electronic batch record along with verification details including the transaction id and stored hash.

You can also call the verification endpoints directly:

GET /anchors

GET /anchors/batch/{id}

GET /batches/{id}/ebr.pdf

Industrial Connectivity

Forge includes lightweight OPC UA and MQTT driver hooks so plant data can be connected without locking the platform to one vendor stack. These drivers are intentionally minimal to keep the architecture understandable and extensible.

In this demonstrator, a driver being marked connected means the endpoint configuration is active, the MES can accept tag mapping definitions for that protocol, and the runtime is ready to bind source values into batch or equipment fields.

See the separate tag-mapping page for example industrial source mappings.
